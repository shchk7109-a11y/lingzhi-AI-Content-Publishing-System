# 灵芝水铺发布系统混合架构设计

## 背景与结论

当前仓库是 `lingzhi-publisher`，定位是灵芝水铺本地发布系统。现有代码已经具备 Electron 桌面应用、内容池、账号表、Bit 指纹浏览器连接、小红书测试发布器等基础，但 Excel 导入、智能匹配、批量调度、互动任务和线上协同还没有形成闭环。

升级方向采用“腾讯云控制台 + 本地 Windows 执行器”的混合架构。腾讯云负责任务中心、审核确认、账号资产台账、批次管理、日志审计和报表；本地 Windows 执行器负责连接 Bit 指纹浏览器、读取本地素材、执行小红书和微信等平台动作，并把截图、日志、结果链接回传到云端。

完整线上化作为后续能力保留，但不作为第一阶段目标。原因是平台自动化依赖已登录浏览器、GUI 会话、指纹浏览器、固定代理、人工接管和素材路径，全部放到云服务器会显著增加稳定性和安全风险。

## 目标

1. 让 `lingzhi-tuwen-pro` 能按照发布系统需要输出标准任务包。
2. 让发布系统可以导入、校验、预览、确认和调度发布任务。
3. 让每个博主 IP、平台账号、Bit Profile、代理 IP 和发布限额形成稳定绑定。
4. 支持小红书发布任务和授权账号互动任务，互动包括评论、收藏、浏览等可审计动作。
5. 所有真实平台操作必须经过人工确认，并保留审计日志与失败截图。
6. 第一阶段优先打通本地执行闭环，第二阶段再接腾讯云控制台。

## 非目标

1. 第一阶段不直接把 Electron 应用整体搬到腾讯云运行。
2. 不绕过平台登录、风控或授权边界。
3. 不在代码、Excel、日志或云端明文保存账号密码、cookie、API key。
4. 不让互动任务自动无确认批量执行。
5. 不要求 `lingzhi-tuwen-pro` 自己理解发布执行细节，只要求它按协议生成任务包。

## 系统分层

### lingzhi-tuwen-pro

内容生成上游。它根据门店、产品、博主人设和平台策略生成小红书、微信朋友圈等文案与配图，并导出标准任务包。它不直接操作平台账号，也不管理 Bit 浏览器。

### 腾讯云控制台

线上控制平面。它保存任务批次、账号资产、审核状态、调度策略、执行节点心跳、任务日志和统计报表。它向本地执行器下发已确认任务，并接收执行结果。

### 本地 Windows 执行器

真实执行平面。它运行在安装了 Bit 指纹浏览器的 Windows 机器上，负责打开对应 Bit Profile，执行发布、评论、收藏、浏览等动作，并把截图、步骤日志、失败原因和结果 URL 回传控制台。

### 平台适配器

每个平台有独立适配器。小红书发布器、微信朋友圈发布器、小红书互动器分别实现自己的页面选择器、步骤、失败识别和截图规则。调度器只认识标准任务，不直接依赖具体页面结构。

## 数据流

1. `lingzhi-tuwen-pro` 导出任务包。
2. 发布系统导入任务包，校验字段、素材目录、账号别名、平台类型和风险等级。
3. 用户在控制台或本地应用中预览批次，确认要执行的发布任务和互动任务。
4. 调度器按账号限额、发布时间窗、平台间隔和风险等级生成执行队列。
5. 本地执行器拉取或接收任务，申请对应 Bit Profile 窗口。
6. 平台适配器执行动作，逐步骤记录状态和截图。
7. 执行器回传成功 URL、失败原因、截图路径、耗时和审计事件。
8. 控制台更新批次状态，失败任务进入隔离队列或人工复核队列。

## 任务包协议 v1

任务包采用目录形式，方便同时携带 Excel 和素材。

```text
batch_YYYYMMDD_HHMM/
  manifest.json
  tasks.xlsx
  media/
    draft_001/
      1.jpg
      2.jpg
    draft_002/
      cover.jpg
```

### manifest.json

```json
{
  "protocol_version": "1.0",
  "batch_id": "batch_20260627_001",
  "source_app": "lingzhi-tuwen-pro",
  "created_at": "2026-06-27T10:00:00+08:00",
  "default_platform": "xiaohongshu",
  "media_root": "media",
  "row_count": 20
}
```

### tasks.xlsx 字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| batch_id | 是 | 批次 ID，必须与 manifest 一致 |
| draft_id | 是 | 内容唯一 ID，也是素材目录匹配键 |
| platform | 是 | `xiaohongshu`、`wechat_moments`、`douyin` |
| action_type | 是 | `publish`、`comment`、`favorite`、`collect`、`browse` |
| blogger_id | 是 | 博主人设或 IP ID |
| account_alias | 是 | 授权账号别名，由发布系统映射到平台账号和 Bit Profile |
| title | 条件必填 | 发布小红书笔记时必填，互动任务可为空 |
| content | 条件必填 | 发布正文，发布任务必填 |
| tags | 否 | 逗号分隔的小红书话题 |
| media_folder | 条件必填 | 发布图文或视频任务必填，值为 `media/draft_id` |
| media_type | 条件必填 | `image`、`video`、`none` |
| target_note_url | 条件必填 | 评论、收藏、浏览任务必填 |
| comment_text | 条件必填 | 评论任务必填 |
| publish_window_start | 否 | 最早执行时间，本地时区 |
| publish_window_end | 否 | 最晚执行时间，本地时区 |
| priority | 否 | 数字越大优先级越高 |
| require_manual_confirm | 是 | 第一阶段固定为 `true` |
| risk_level | 是 | `low`、`medium`、`high` |
| remark | 否 | 人工备注 |

### 字段规则

`draft_id` 必须唯一。`account_alias` 不能直接写真实账号密码或 cookie。发布任务必须包含 `title`、`content` 和素材信息。评论任务必须包含 `target_note_url` 和 `comment_text`。收藏和浏览任务必须包含 `target_note_url`。所有任务默认待人工确认。

## 账号资产模型

账号资产以授权账号为中心，不以单次任务为中心。

| 字段 | 说明 |
| --- | --- |
| account_alias | 发布系统内部别名 |
| platform | 平台 |
| blogger_id | 对应博主 IP 或人设 |
| bit_profile_id | Bit 指纹浏览器 Profile ID |
| proxy_profile | 固定代理或代理池配置引用 |
| account_level | 新号、成长期、成熟号 |
| daily_publish_limit | 每日发布上限 |
| daily_interaction_limit | 每日互动上限 |
| status | 活跃、暂停、异常、封禁 |
| last_health_check_at | 最近登录态或可用性检查时间 |

发布系统导入任务时只接受 `account_alias`，再由账号资产表映射到具体 Bit Profile 和代理配置。这样可以让 `lingzhi-tuwen-pro` 不接触敏感账号细节。

## 互动边界

小红书互动只允许自有或明确授权账号执行。系统必须强制人工确认、日志审计和失败隔离。

互动任务规则：

1. `comment`、`favorite`、`collect`、`browse` 都是独立任务类型。
2. 每条互动任务必须有目标 URL。
3. 评论任务必须保存最终评论文本。
4. 同一账号执行互动任务需要随机间隔和每日上限。
5. 失败只隔离当前账号或当前目标，不中断整个批次。
6. 审计日志记录账号别名、目标 URL、动作、时间、结果、截图和错误原因。
7. 高风险任务必须二次确认，例如大量评论、重复评论、短时间密集收藏。

## 错误处理

任务失败按原因分类：

| 类型 | 处理 |
| --- | --- |
| 登录失效 | 暂停账号，进入人工处理 |
| Bit Profile 打不开 | 隔离账号，记录 Bit 错误 |
| 页面结构变化 | 标记适配器异常，停止同类任务 |
| 素材缺失 | 标记任务数据错误，不重试 |
| 平台限流 | 暂停账号并延后调度 |
| 网络超时 | 允许有限重试 |
| 用户取消 | 标记人工取消 |

每次失败都保存截图和最后一步状态。连续失败达到阈值后，调度器停止该账号后续任务。

## 安全与合规

1. 云端不保存账号密码、cookie 或明文代理密码。
2. 本地执行器只保存必要的 Bit Profile 引用和加密后的本地配置。
3. 日志中脱敏账号、代理、手机号、token 和 cookie。
4. 所有批量发布和互动动作默认需要人工确认。
5. 所有任务都有审计记录，支持追溯到批次、账号、素材和操作人。
6. 健康数据或用户个人数据不能明文发送到境外服务。

## 阶段计划

### Phase 0：协议与边界

产出任务包协议 v1、账号资产模型、发布任务和互动任务边界。`lingzhi-tuwen-pro` 按协议调整导出格式。发布系统只做导入校验和预览设计，不接平台。

### Phase 1：本地执行闭环

完成本地 Electron 应用的任务包导入、素材匹配、账号映射、人工确认、单账号小红书发布、日志截图和失败分类。目标是用一个授权测试账号跑通端到端流程。

### Phase 2：批量调度与互动

完成匹配确认、任务队列、窗口池调度、账号限额、发布间隔、互动任务、失败隔离和重试策略。目标是小规模多账号稳定运行。

### Phase 3：腾讯云控制台

新增云端任务中心、批次审核、账号资产台账、执行节点心跳、结果回传和报表。目标是云端管控、本地执行。

### Phase 4：云端执行节点评估

在混合架构稳定后，评估是否增加腾讯云 Windows 执行节点。该能力用于补充执行容量，不替代本地执行器作为第一阶段主路径。

## 验收标准

1. 发布系统能导入标准任务包，并给出明确校验报告。
2. 用户能在执行前看到每条任务的账号、平台、动作、文案、素材和风险等级。
3. 单个授权小红书账号能完成发布任务，并保存结果 URL、步骤日志和截图。
4. 互动任务默认待确认，执行后能在审计日志中追溯。
5. 任一账号失败不会中断整个批次。
6. 账号密码、cookie、代理密钥不会出现在日志、Excel 或云端明文字段中。
7. 腾讯云控制台上线前，本地执行器已经可以独立完成导入、确认、执行和回传本地日志。

## 待确认决策

1. 第一版任务包是否只支持小红书发布和小红书互动，微信朋友圈放到 Phase 2。
2. `account_alias` 是否由发布系统统一生成并回填给 `lingzhi-tuwen-pro` 使用。
3. 云端控制台是否采用独立 Web 项目，避免把 Electron 主进程逻辑直接搬到云端。
