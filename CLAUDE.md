## 二、核心工作原则（来自 gstack ETHOS）

### 原则 1：把湖烧干（Boil the Lake）
AI 辅助编码使"完整实现"的边际成本接近零。当完整方案比捷径只多 70 行代码时，**永远选择完整方案**。

**禁止以下思维：**
- ❌ "先跳过测试，后续 PR 补" → 测试是最便宜的湖，现在就写
- ❌ "这个边界情况概率低，先不处理" → 处理它只需几分钟
- ❌ "简化版先上线" → 除非是刻意的 MVP 决策，否则做完整版

### 原则 2：先搜索，再构建（Search Before Building）
遇到不熟悉的模式、第三方库、部署配置时，**先搜索确认当前最佳实践**，再动手。

三层知识体系：
1. **经过验证的方案** → 直接用，但要问"这个假设在我们场景下还成立吗？"
2. **新兴流行方案** → 搜索后批判性采纳，人群可能是错的
3. **第一性原理** → 针对灵芝水铺的具体问题做原创推理，这是最有价值的

### 原则 3：Iron Law — 先调查，再修复
**禁止在未完成根因分析前提交任何修复。**

调试标准流程：
1. 梳理数据流，确认问题边界
2. 列出 3 个可能原因（按概率排序）
3. 逐一验证，找到根因
4. 修复后写回归测试
5. 如果连续 3 次修复都失败 → 停止，上报给用户，不要继续猜

## 三、开发流水线（Sprint 流程）

```
思考 → 计划 → 构建 → 自查 → 测试 → 提交 → 复盘
```

每个阶段的具体要求：

### 阶段 1：思考（Think）
收到新需求时，必须先回答：
- 这个需求解决的真实痛点是什么？
- 有没有更简单的方案达到同样效果？
- 会影响哪些现有模块？（列出来）
- 预计影响哪些文件？

### 阶段 2：计划（Plan）
动手前输出：
- [ ] 要修改/新建的文件列表
- [ ] 数据流变化说明
- [ ] 边界情况清单
- [ ] 需要同步更新的文档

**等待用户确认计划后再开始编码。**

### 阶段 3：构建（Build）
- 每完成一个逻辑单元就做一次 git commit（原子提交）
- commit message 格式：`feat/fix/refactor(模块): 具体描述`
- 中文注释，英文代码
- 遇到不确定的第三方 API → 先查文档，不要凭记忆

### 阶段 4：自查（Review）
提交前必须自查：

**代码质量：**
- [ ] 是否有未处理的 Promise rejection？
- [ ] 是否有硬编码的敏感信息（API key、密码）？
- [ ] 错误处理是否覆盖网络超时、API 限流情况？
- [ ] 类型定义是否完整（TypeScript 项目）？

**业务逻辑：**
- [ ] 是否覆盖了空数据、异常数据的情况？
- [ ] 并发场景下是否存在竞争条件？
- [ ] 国内网络环境下是否需要代理配置？

### 阶段 5：测试（Test）
- 每个新功能必须附带测试
- 修复 bug 必须附带回归测试
- 测试覆盖率下降时必须提醒用户

### 阶段 6：提交与部署（Ship）

**Claude Code 的职责边界（重要）：**
- ✅ 可以：git add / commit / push 到 GitHub
- ✅ 可以：生成 Railway 部署的详细操作指令
- ❌ 不可以：直接操作 Railway 控制台（交给 Manus）
- ❌ 不可以：直接 merge main 分支（需用户确认）

**Railway 部署指令模板（生成给 Manus 执行）：**
```
Manus 部署任务：
1. 登录 Railway 控制台
2. 进入项目 [项目名]
3. 确认最新 commit 已同步：[commit hash]
4. 触发重新部署
5. 等待健康检查通过（访问 /api/health）
6. 确认生产环境环境变量完整（见下方列表）
必需环境变量：[列出]
```

### 阶段 7：复盘（Retro）
每周或每个功能里程碑后，输出：
- 本周提交数、净增代码行
- 完成的主要功能
- 遗留的技术债
- 下周优先级

## 四、中国开发环境专项规范

### 网络代理
- 所有访问境外 API 的请求，默认走 V2RayN 代理（本地 127.0.0.1:10809）
- 国内 API relay 地址优先于官方直连
- npm 包安装：`npm config set registry https://registry.npmmirror.com`

### API 配置（灵芝健康APP）
```
Gemini → 国内 API relay（多模态任务）
DeepSeek → 直连（文字推理任务）
```
永远不要在代码里硬编码 API key，使用 `.env.local`。

### Railway 部署注意
- 环境变量改动后必须重新部署
- Railway 的 Node.js 版本与本地一致（当前：20.x）
- 数据库迁移脚本在部署前必须在本地验证通过

### 小红书/抖音/微信平台（发布系统）
- Bit 指纹浏览器窗口池并发数：默认不超过 10（避免平台风控）
- 账号操作间隔：随机 2-5 秒（模拟人工）
- 批量任务失败时：单账号隔离处理，不要中断整体任务

## 五、各项目专属规范

### 5.1 灵芝健康APP（Next.js）

**七知识库架构（不得随意修改）：**
- KB1: 九种体质（王琦标准）
- KB2: 8个灵芝产品 + 体质映射
- KB3: 东西方人群差异
- KB4: 106种药食同源
- KB5: 基因营养关联（含 MTHFR）
- KB6: 节气养生
- KB7: 22058个已批准保健食品数据集

**核心约束：**
- 灵芝作为战略性成分，在 KB4 和 KB7 中均有交叉引用，修改时两处同步
- AI 健康建议必须附带免责声明
- 产品推荐逻辑改动必须经过体质匹配测试

**性能要求：**
- 流式输出延迟 < 500ms（首字）
- 雷达图渲染 < 200ms
- 页面 LCP < 2.5s

### 5.2 多平台智能发布系统 v2.0（Electron）

**模块边界（不得跨模块直接调用）：**
```
CustomerManager → WindowPool → Publisher
                              ↗
InteractionEngine             
                              ↘
WeChatBatchSender → Publisher
```

**当前待完成：**
- [ ] Excel 上传后自动提取笔记标题用于评论生成
- 实现方式：读取 Excel 第一列/标题行 → 传入 AI 评论生成器

**安全约束：**
- 账号 cookie/密码不得写入日志
- 批量操作前必须有用户二次确认
- Bit 浏览器窗口异常关闭时自动从池中移除

### 5.3 灵芝视频工作室（后端）

**六步流水线（顺序不得变更）：**
```
1. 脚本生成 → 2. 素材搜集 → 3. 语音合成
→ 4. 视频合成 → 5. 字幕生成 → 6. 最终导出
```

每步都要有状态持久化，支持断点续传。

## 六、禁止事项（Hard Rules）

1. **禁止** 直接修改 main 分支（必须通过 PR 或用户明确授权）
2. **禁止** 在代码中硬编码任何密钥、密码、Token
3. **禁止** 删除测试文件（即使测试在报错）
4. **禁止** 在未通知用户的情况下变更数据库 schema
5. **禁止** 在调查未完成时提交"临时修复"
6. **禁止** 降低 TypeScript 严格模式（noImplicitAny 等）
7. **禁止** 向境外服务发送用户的健康数据明文

---

## 七、当 Claude Code 遇到不确定情况时

**遇到以下情况，停下来问用户，不要自行决定：**
- 需要修改超过 5 个文件的重构
- 数据库结构变更
- 第三方服务配置变更（Railway、Supabase 等）
- 删除任何现有功能
- 发现与现有业务逻辑冲突的需求

**遇到以下情况，自行处理但事后说明：**
- 新增工具函数
- 修复明确的 bug（已完成调查）
- 更新注释和文档
- 添加测试用例
